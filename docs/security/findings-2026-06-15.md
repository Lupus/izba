# izba Security Findings — 2026-06-15 (first pass)

> **Method:** multi-agent static/manual attack-surface survey across all six
> trust boundaries, followed by independent re-verification of every HIGH
> candidate against the source. See [methodology.md](methodology.md).
>
> **Maturity caveat — read first:** these are **code-confirmed candidate
> findings**, not weaponized exploits. "Repro: code-confirmed" means the
> orchestrator independently re-read the cited lines and the described code
> behavior is real; it does **not** mean an end-to-end PoC was built. Per the
> methodology, HIGH findings owe a PoC (fuzz/exploit) in the next pass before
> they are "Confirmed-exploited." Severities are provisional CVSS-style
> estimates under the guest-hostile environment (threat-model A1).
>
> Scope caveat: `mitm.rs` is headed "M5 SPIKE" but `build_mitm_runtime`
> **does** wire a live `MitmRuntime` into production `DaemonDeps`, and the
> router routes enforcing :80/:443 through it — so the MITM path is **live for
> any sandbox created with `--policy`**. Bare (no-`--policy`) sandboxes are
> AllowAll/non-enforcing with transparent direct-dial.

## HIGH

### F-01 — Inverted egress is an unauthenticated SSRF-capable proxy under the default AllowAll policy
- **Boundary:** B3 (guest→izbad egress). **Repro:** code-confirmed (router.rs:56-145).
- A bare/M1 sandbox runs `AllowAll` (non-enforcing). `decide_tier2` returns
  Allow for **any** raw IP — the `is_private` guard at router.rs:263 is applied
  *only* when `policy.enforces()`. The hostile guest sends
  `StreamOpen::TcpConnect{addr,port}` and izbad dials it **from the host's
  network namespace** — `127.0.0.1:*`, LAN, cloud metadata `169.254.169.254`.
  The dummy0/nft "deny" only constrains the guest's own datapath; the proxy
  runs on the host. There is no default-deny unless a `--policy` is attached.
- **Fix:** apply the private/loopback/link-local/metadata denylist
  **unconditionally** (independent of `enforces()`); make default-deny the
  baseline posture for *all* sandboxes, with AllowAll an explicit opt-in.

### F-02 — MITM never checks ClientHello SNI == HTTP Host (allow-list bypass / domain fronting)
- **Boundary:** B5. **Repro:** code-confirmed (mitm.rs:233 leaf-by-SNI;
  mitm.rs:401-408,461,479 policy/upstream-by-Host; mitm_runtime.rs:230 dial-by-OrigDst).
- The leaf is minted for the **ClientHello SNI**; the policy verdict, the
  upstream TLS server-name, and the audited host are taken from the **decrypted
  HTTP Host header**. Nothing asserts SNI == Host. The upstream socket is dialed
  at the guest-chosen `OrigDst` IP while the cert is verified against `Host`.
  A guest can present `SNI=anything`, `Host=allowed.example.com`, `OrigDst=` an
  IP that serves a valid cert for the allowed Host — passing policy while izbad
  TCP-connects where the guest chose.
- **Fix:** require SNI == Host (or reconcile + re-verify); pin the upstream dial
  to the policy-approved name's resolved address, not guest `OrigDst`.

### F-03 — Only the first HTTP request on a kept-alive MITM connection is policy-checked
- **Boundary:** B5. **Repro:** code-confirmed (mitm.rs:458-488).
- `read_request_head` runs **exactly once**; after Allow, `pump_bidirectional`
  copies all remaining bytes verbatim. HTTP/1.1 keep-alive lets the guest send
  `GET allowed.com` to pass, then reuse the same TLS+TCP session for requests
  with arbitrary `Host`/path. The audit log records only the first request, so
  `izba netlog` systematically under-reports.
- **Fix:** parse and policy-check **every** request on the connection (full
  HTTP/1.1 framing), or disable keep-alive (force `Connection: close`) so one
  connection = one checked request.

### F-06 — VMM and virtiofsd run unjailed/unconfined as the full host user
- **Boundary:** A2 (containment). **Repro:** code-confirmed (cloud_hypervisor.rs:131
  via `spawn_detached`; procmgr/unix.rs:23).
- cloud-hypervisor, openvmm and virtiofsd are launched with **no user/namespace
  drop, no seccomp, no chroot, no resource jail** — unlike Firecracker's jailer.
  A VM-escape (B-VIRTIO) or virtiofs bug therefore yields the **invoking host
  user's** full privileges. For a product whose purpose is sandboxing, the
  host-side VM process is the last line of defense and it is unconfined.
- **Fix:** a jailer-equivalent per sandbox: dedicated uid/gid, mount + pid + net
  namespaces, a seccomp profile, cgroup limits; minimally, drop privileges and
  seccomp-confine the VMM and virtiofsd processes.
- **Status update (2026-06-16) — Windows VMM: MITIGATED** (branch/PR
  `windows-vmm-jailer`; spec
  [docs/superpowers/specs/2026-06-16-windows-vmm-jailer-design.md](../superpowers/specs/2026-06-16-windows-vmm-jailer-design.md),
  plan
  [docs/superpowers/plans/2026-06-16-windows-vmm-jailer.md](../superpowers/plans/2026-06-16-windows-vmm-jailer.md)).
  The OpenVMM process now launches **confined by default**: a restricted token
  (`DISABLE_MAX_PRIVILEGE`, all privileges dropped) + **Low integrity** + a
  best-effort resource job (no kill-on-close) + creation-time process
  mitigations. It **fails closed** (errors rather than running unconfined)
  unless the user passes `--allow-unconfined` (which warns loudly); `izba status
  NAME` reports the confinement honestly. Proven in the `windows-whp` e2e via a
  differential `confine_probe` PoC + a live-status assertion.
- **Workspace integrity model:** the confined VMM runs at Low IL, so MIC
  no-write-up means it can only write Low-or-below objects. izba therefore
  Low-labels exactly the surface the VMM must write — the per-sandbox scratch dir
  (console.log/rw.img/vsock) **and** the virtiofs workspace share (the user's
  project dir; the guest writes `/workspace` through the VMM's in-process
  virtiofs). The label is inheritable and **probe-proven** to propagate to
  plain-created children, and is **restored to ~Medium on teardown** (graceful
  stop, force-remove, the daemon-adoption stale-state sweep that reconciles
  orphaned VMMs, AND the launch error paths — since relabel precedes state.json, a
  failed confined start is restored explicitly so the user's dir is never stranded
  at Low). A missed restore is benign (Medium tools write *down* to a Low dir
  freely). Everything *outside* the shared workspace stays Medium and so is
  no-write-up-protected from the Low-IL VMM — that is the integrity boundary.
  **Accepted residuals of integrity-relabelling** (all benign for the boundary,
  surfaced by adversarial review, code-documented; each closed by the
  dedicated-account hardening tier which never relabels the user's dir): restore
  is approximate (re-asserts explicit Medium, not the exact prior SACL);
  atomic-rename-in saves from outside the tree keep a non-Low label the guest
  can't write (narrow); VMM-created files keep their explicit Low label after
  teardown (workspace left with some Low files); and teardown keys on the tracked
  parent pid, so an untracked surviving `openvmm vm` worker is a pre-existing
  liveness gap.
- **Accepted residual (live-validated 2026-06-16):** **no read-confinement on
  Windows.** Restricting-SID shaping is infeasible for this single-token model
  (process init fails `0xC0000142`, even native cmd.exe; Chromium's two-token
  warmup is unavailable to a third-party VMM). The token therefore gives
  **integrity** protection (no write-up to the user's files, no privilege
  escalation, resource-capped) but a Low-IL VMM **can still
  read** the user's files outside the share. (Child-process creation is **not**
  blocked — OpenVMM forks a worker and the Windows primitive is all-or-nothing;
  children inherit the restricted token + Low IL, so they are deprivileged but
  not prevented from spawning.) Read-confinement needs a **distinct principal**:
  AppContainer (probe-proven to break WHP → ruled out) or a **dedicated
  low-privilege local account** for the VMM (one-time admin). The latter is the
  documented **future hardening tier** (`izba install --harden`): run the VMM as
  an `izba-vmm` account in Hyper-V Administrators with the workspace ACL-granted
  per run → read **and** write confinement, closing this residual (and removing
  the workspace integrity-relabel). Probe-gated on (i) WHP under a separate
  account and (ii) cross-account ACL plumbing. **Accepted for now** (integrity
  floor shipped; account ceiling documented) — rationale in
  [windows-vmm-jailer-chromium-reference.md](windows-vmm-jailer-chromium-reference.md)
  and the spec above.
- **Still OPEN under F-06:** the **Linux** host-side VMM jailer
  (cloud-hypervisor/openvmm — no namespace/seccomp/cgroup drop) and
  **virtiofsd** confinement (the latter also tracked as **F-07**). F-06 is thus
  **partially mitigated**, not closed.

### F-07 — virtiofsd launched with `--sandbox none`
- **Boundary:** B4 / A2. **Repro:** code-confirmed (cloud_hypervisor.rs:62-64).
- virtiofsd — the component directly parsing guest FUSE requests against the
  shared host dir (which **is the user's real project directory**) — is started
  with `--sandbox none`, disabling its own namespace + seccomp self-confinement.
  A path-traversal/symlink or memory bug in virtiofsd then operates with the
  full host-user filesystem view (cf. Kata CVE-2026-47243).
- **Fix:** `--sandbox chroot` (or run virtiofsd inside a user namespace);
  combine with F-06 so a virtiofsd compromise is contained regardless.

### F-08 — Host-side `cp` unpack of a guest tar has no openat2 confinement (guest→host arbitrary write)
- **Boundary:** B-CP. **Repro:** code-confirmed (cp.rs:119-149).
- `copy_from_guest` unpacks an **untrusted guest-produced** archive with
  `entry.unpack(&dest_path)` + `create_dir_all(parent)`, trusting the `tar`
  crate alone. The guest-side receiver (`izba-init/tarfs.rs`) is carefully
  hardened (openat2 `RESOLVE_IN_ROOT`, lexical `..` rejection, final-symlink
  replacement) — the host side has **none** of that. `rewrite_top` rewrites only
  the first path component, so traversal components later in an entry name pass
  through the join; the classic two-step symlink escape (entry A = symlink
  `x → /tmp/evil` or `../../..`, entry B = file `x/payload`) depends entirely on
  the exact `tar` 0.4.x hardening with no second line of defense. This is the
  highest-value lead: the guest is the attacker and the write lands on the host.
- **Fix:** mirror the guest-side hardening on the host — openat2
  `RESOLVE_IN_ROOT` / `O_NOFOLLOW` per entry, reject `..`/absolute, refuse to
  follow symlinks during extraction; add a differential test against tarfs.rs.

## MEDIUM

### F-04 — Egress audit log: no integrity protection, lossy, under-reports
- **Repro:** code-confirmed (audit.rs:185-201). Plaintext JSONL, append,
  **all write errors silently swallowed**, no hash-chain/fsync; host-writable.
  Combined with F-03 it under-reports what a hostile guest actually sent. Audit
  is observability, not a control. **Fix:** append-only + hash-chain (or signed)
  records; surface write failures; log every checked request.

### F-05 — DNS-snoop trusts attacker-influenced answers as the tier-2 allow-list key; DNS forwarder is an open exfil channel
- **Repro:** code-confirmed (router.rs:70-76 port-53 short-circuit *before any
  policy*; dns_snoop.rs:36-57; router.rs:226-307 decide_tier2). All DNS (UDP +
  TCP:53) is unconditionally forwarded to the host resolver, even for an
  enforcing sandbox — a covert C2/exfil side-channel (data in QNAMEs) with no
  rate limit. `extract_a_aaaa` then records `(allowed-name → resolved-IP)` from
  guest-chosen queries; a guest that influences resolution (own domain / low-TTL
  rebind) can plant `allowed-name → attacker-IP` and dial it on a non-HTTP port
  to pass tier-2. The 60 s TTL floor *widens* the rebinding window.
  **Fix:** subject DNS to per-sandbox policy + rate limits; do not treat snoop
  data as authorization (it is a heuristic — threat-model §8); for enforcing
  sandboxes resolve allow-listed names *in izbad* and pin dials to that result.

### F-09 — izbad control socket has no authentication or peer-credential check
- **Repro:** code-confirmed (server.rs:192-197; transport.rs:28-40). The hello
  is read and **ignored for auth**; no `SO_PEERCRED`/`getpeereid`, no token. Sole
  gate is the 0700 daemon dir. Any process that can open the socket can
  Create/Start/Stop/Rm sandboxes, `GuestRpc`-exec inside guests, `OpenStream`-
  splice a shell into any sandbox, `PortPublish`, and `Shutdown`. **Fix:** add an
  `SO_PEERCRED` uid match (reject non-owner) as defense-in-depth even with 0700.

### F-10 — `OpenStream` splices a client-chosen, daemon-unparsed `StreamOpen` to the guest
- **Repro:** code-confirmed (server.rs:221-227). After `OpenStream{name}` the
  daemon liveness-gates the name, replies Ok, then splices raw bytes; the client
  sends an arbitrary `StreamOpen` (Attach/TcpDial/TarExtract/TarCreate) the
  daemon never inspects. With F-09 (no auth), any local process can drive
  exec/cp/relay into any sandbox. Authorization is "can you reach the socket."
  **Fix:** depends on F-09; consider parsing/authorizing the spliced intent.

### F-12 — 16 MiB per-frame allocation on a length prefix before content validation (alloc-amplification DoS)
- **Repro:** code-confirmed (codec.rs:41). `read_frame` checks `len ≤ 16 MiB`
  then `vec![0u8; len]`. izbad spawns a thread per egress/control connection with
  no connection cap; N concurrent connections each force a 16 MiB alloc → memory
  exhaustion. A guest can open many vsock-1027 connections. **Fix:** per-peer
  connection ceiling + streaming/bounded reads; lower the cap per message type.

### F-13 — Unbounded thread-per-connection on guest-initiated egress
- **Repro:** code-confirmed (egress/mod.rs:159-172). The accept loop spawns an
  unbounded detached thread per vsock-1027 connection; each dials outbound and
  (MITM) registers a loopback ephemeral port. A tight connection loop exhausts
  host threads/sockets/ports. **Fix:** bounded worker pool + per-sandbox
  concurrency limit + backpressure.

### F-15 — Per-sandbox dirs and the data root created with default umask, not 0700
- **Repro:** code-confirmed (sandbox.rs:164, paths.rs:26). Only `daemon/` and
  `ca/` are explicitly 0700. The data root, `sandboxes/`, run_dir (holds
  `vsock.sock`) and logs are created under the process umask (commonly 0022 →
  world-traversable). On a multi-user host this leaks layout, console logs, and
  the control-socket path. **Fix:** chmod the whole tree 0700 consistently.

### F-17 — No decompression-bomb / file-count / total-size cap when staging+flattening OCI layers
- **Repro:** code-confirmed (image/flatten.rs:40-65). `std::io::copy(GzDecoder,
  tmp)` per layer with no ceiling; an unbounded entry count walked into an
  in-memory `BTreeMap`. A malicious image (gigabytes-of-zeros gzip, or millions
  of tiny entries) fills host disk + exhausts memory — DoS from `izba run
  <evil-image>`. **Fix:** cap decompressed size, entry count, and total bytes;
  stream rather than buffer the path map.

### F-22 — No cargo-audit / cargo-deny gate in CI
- **Repro:** code-confirmed (no `deny.toml`/`audit.toml`; no RUSTSEC step in any
  `.github/workflows`). The graph has a large net/crypto/TLS surface (reqwest,
  hyper, rustls, ring, aws-lc-rs, regorus, oci-client, vt100, portable-pty,
  vsock). A known-vulnerable transitive dep can ship silently. **Fix:** add
  `cargo-deny` (advisories + bans + licenses) + `cargo-audit` as a green-before-
  commit gate (methodology §C).

### F-23 — CA leaf certs never expire; CA omits validity window and pathlen
- **Repro:** code-confirmed (mitm.rs:65-76, 168-189). Neither CA params nor
  per-host leaves set `not_before`/`not_after`; CA is `BasicConstraints::
  Unconstrained` (no `pathlen:0`), no name-constraints, no short-lived rotation.
  If `ca.key` leaks, every guest trusting `ca.pem` accepts attacker leaves for
  **any** host indefinitely — no CRL/OCSP despite declared `CrlSign`. **Fix:**
  short leaf validity + rotation; `pathlen:0`; name-constraints to scope blast
  radius; plan revocation.

## LOW

- **F-16 — CA key is long-lived plaintext PKCS#8, readable by any same-user
  process; never rotates.** (ca.rs:19-73; perms 0600/0700 are correct.) Residual
  scope risk: any code as the daemon user can mint universally-trusted leaves.
  *Fix:* OS keystore / passphrase / rotation where feasible.
- **F-18 — Guest can accumulate zombie PIDs until poweroff.** (izba-init
  main.rs:172, exec.rs:64.) PID1 doesn't reap(-1) by design; guest-local DoS,
  not host compromise. *Fix:* bound per-exec retention / reap policy.
- **F-24 — CH backend does not reject commas in disk/workspace paths**
  (cloud_hypervisor.rs:86) — asymmetric with openvmm's `reject_commas()`.
  Option-injection via path contents (no shell; argv is passed directly).
  *Fix:* mirror the openvmm guard.
- **F-25 — openvmm binary pinned to an expiring (~90-day) GitHub CI artifact**
  (hack/fetch-openvmm.sh:13) rather than a signed/reproducible release. sha256
  pin is good but provenance is fragile. *Fix:* track a stable upstream release.

## INFO / By-design (record, don't "fix" casually)

- **F-11 — Image identity rests on a registry manifest digest with no
  signature/provenance; anonymous-only auth** (image/pull.rs:41-63). Layers are
  verified against *that* manifest (good intra-manifest integrity) but there is
  no cosign/notation. A malicious public image becomes the guest root.
  *Consider:* optional signature verification / digest pinning enforcement.
- **F-14 — `state.json`/`config.json` fields (image_digest, workspace, ports)
  trusted as written** (sandbox.rs:335,365). Liveness is correctly re-derived
  (pid+starttime — a strength), but a local user who can write the sandbox dir
  (see F-15) could repoint workspace/image. Gated by F-15.
- **F-19 — Guest-supplied uid/gid applied verbatim in exec** (izba-init
  exec.rs:219). Intended: the control plane is trusted (izbad). Noted for the
  future multi-tenant mesh, where the control plane may be exposed more broadly.
- **F-21 — The in-guest nft/dummy0 "structural deny" is not a containment
  boundary.** A hostile guest root can flush nft and talk AF_VSOCK 1027 directly
  (net.rs, egress.rs). Enforcement that matters is 100% izbad-side. Already
  documented as cooperative — flagged so no future reasoning treats it as a
  control. (threat-model §8.)

## Mitigated on verification (would-be false positive)

- **F-26 — "Daemon name strings drive filesystem paths with no validation."**
  Downgraded: `validate_name` (`[a-z0-9][a-z0-9_.-]*`, max 64, rejects
  `..`/`/`/uppercase/leading-dot) is called at every sandbox-name entry point
  (sandbox.rs:159,362,501,514,634). Path traversal via name is **blocked**.
  Residual hardening only: validate at the protocol-dispatch layer too for
  defense-in-depth. *Kept as a record of the adversarial-verification gate
  doing its job — methodology §B.5.*

## Strengths worth recording (a balanced report)

- **No shell** is ever used to launch the VMM — argv passed element-by-element
  via `std::process::Command` (no shell-metachar injection).
- **PID-reuse-safe liveness** (pid + `/proc/<pid>/stat` starttime; zombies =
  dead) — genuinely well done.
- **Egress MITM fails *closed*** for enforcing sandboxes when the runtime is
  unavailable (router.rs:96-124, well-tested) — the right failure direction.
- **CA private key never enters a guest** — only the public `ca.pem` is shared
  read-only; key dir/file are 0700/0600.
- **cloud-hypervisor, virtiofsd, and the vendored static nft are sha256-pinned**
  (Alpine builder by digest, netfilter tarballs by hash).
- **Sandbox names validated** path-traversal-safe; image digests verified
  against the manifest.
