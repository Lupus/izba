# M2.1 — Interactive firewall: hot-reload + port-aware policy + actionable netlog

> Design doc. Status: **approved (brainstorm), pending spec review**. Date 2026-06-15.
> Builds directly on [M2 agent firewall](2026-06-14-m2-agent-firewall-merged-design.md)
> (MITM L7 + two-tier policy + `izba netlog` + per-sandbox `--policy`). This is the
> usability follow-up that makes the M2 firewall *operable* — edit policy without a
> restart, see traffic as an actionable summary, and close a real port-scoping
> loophole. It also defines what the desktop app's **P4 (ports + firewall/netlog)**
> milestone actually is. Roadmap: a tightening of M2 (call it **M2.1**); does not
> touch M3/M4 sequencing.

## Why

M2 shipped a working firewall but three things make it awkward to actually use, and
one is a security gap:

1. **Editing policy requires a sandbox restart.** A sandbox's `--policy` file is read
   once when the egress listener is armed (`EgressManager::ensure_listening` →
   `resolve_policy`, `daemon/egress/mod.rs:150`); the accept thread captures that
   `Arc<dyn Policy>` for its lifetime. Changing the allow-list means stop/start —
   ugly, and it kills the agent's other live work.
2. **`izba netlog` is a flat text tail.** One JSON line per connection attempt
   (`daemon/egress/audit.rs`), so a chatty agent floods it; there's no per-endpoint
   summary, no counts, and no way to act on what you see.
3. **No way to allow/deny a destination interactively** — the whole "see a blocked
   connection, click allow" loop sbx has is absent.
4. **Port loophole (security gap).** The Rego allow-list matches `dest_name in
   allow-list` with **no port predicate** (`egress.rego:33-43`; `input.port` is
   passed but never read). For an enforcing sandbox, non-HTTP ports route through
   `decide_tier2` (`router.rs:226`), which allows the flow if *any* DNS-snooped FQDN
   for the destination IP is allow-listed — also port-agnostic. So allowing
   `github.com` for HTTPS *also* authorizes `github.com:22`, `:3306`, and any other
   port the guest can resolve to it — a real exfiltration channel, and there is no
   YAML grammar to scope a host to specific ports.

This design fixes all four as one coherent change: **make the M2 firewall
interactive**.

## Non-goals

- HTTP method/path L7 rules (the M5 horizon already stubbed in `egress.rego`); we
  stay at host+port granularity.
- A runtime policy *API* (mutate rules over a socket). Locked decision stands:
  policy lives in `policy.yaml`; changes apply via a **reload verb**. The app/CLI
  edit the file and ask the daemon to reload — they do not push rule objects.
- Credential injection, east–west mesh, projects (M4/M5).
- IP-literal allow rules. The allow-list is name-based by design (the SSRF /
  DNS-rebind guard). A raw-IP destination with no resolved name stays deny-only.

## Locked decisions (from the brainstorm)

- **Port default:** an allow-list entry with no explicit `ports` authorizes **80 and
  443 only**. Other ports require an explicit `ports: [...]`. This closes the
  loophole by default and is the intended behavior change (today an allowed domain
  is reachable on any port via tier-2).
- **Explicit ports replace the default**, they do not augment it: `ports: [5432]`
  means *only* 5432. A host wanted on web + a custom port lists both: `[443, 5432]`.
- **Bare (allow-all) sandboxes** get an **"Enable firewall" CTA** seeded from
  currently-observed allowed endpoints (so enabling the firewall does not instantly
  break the agent). Per-row allow/block actions are live only once a sandbox is
  enforcing.
- **Aggregation key is `(host-or-ip, port)`**, one row per endpoint, showing the
  current/latest verdict plus allow & deny counts; sorted by `last_seen` descending
  (fresh hits bump to the top).
- **Aggregation and policy-editing live in core** (shared, tested Rust), consumed by
  both the CLI and the Tauri app. No parsing/aggregation logic in TypeScript.
- **Edit verbs/buttons auto-reload** (write file + `ReloadPolicy` in one step);
  `izba policy reload` is the manual escape hatch after hand-editing.

## Architecture: three steps, core-first

Each step is independently shippable and gets its own TDD implementation plan.

```
Step 1  core   policy hot-reload          (swappable cell + ReloadPolicy RPC)
Step 2  core   port-aware grammar + rego  (close the loophole)
Step 3  core   shared helpers             (audit::aggregate + policy_edit)
        + CLI  izba policy {show,allow,block,reload}, izba netlog --summary
        + app  P4: actionable netlog tab + policy editor + enable-firewall CTA
```

## Step 1 — Policy hot-reload

### Current state

`EgressManager` (`daemon/egress/mod.rs`) owns one `EgressSlot` per sandbox:

```rust
struct EgressSlot { stop: Arc<AtomicBool>, thread: JoinHandle<()> }
```

`ensure_listening` resolves the policy **once** and the accept-thread closure captures
it:

```rust
let policy = resolve_policy(&self.policy, paths, name); // Arc<dyn Policy>, fixed
// ... per connection: let policy = Arc::clone(&policy); router::handle_conn(..)
```

Per-connection the `Arc` is cloned into `router::handle_conn`, and for tier-1 it
travels into the shared `MitmRuntime` per flow via `register(src_port, dst, policy)`.

### Change

Store the policy behind a swappable cell read **per connection** rather than captured
once:

```rust
struct EgressSlot {
    stop: Arc<AtomicBool>,
    thread: JoinHandle<()>,
    policy: Arc<PolicyCell>,   // shared with the accept thread
}

// PolicyCell wraps an atomically-swappable Arc<dyn Policy>.
// Implementation: Arc<Mutex<Arc<dyn Policy>>> (no new dependency) with
// `load()` (clone the inner Arc under a short lock) and `store(new)`.
// arc-swap is a fine alternative if we want lock-free loads; Mutex is
// adequate (the lock is held only for an Arc clone/replace, never across I/O).
```

The accept loop calls `slot_policy.load()` for each accepted connection (instead of
`Arc::clone` of a captured value), so a later `store()` takes effect on the next
connection. In-flight connections keep the snapshot they already cloned — this is the
**new-flows-only** semantics already decided for `izba project reload`.

### Reload entry point

```rust
impl EgressManager {
    /// Re-resolve `name`'s policy from disk and swap it into the live slot.
    /// No-op (Ok) if the sandbox has no live egress slot — the file on disk is
    /// already what the next start will read.
    pub fn reload_policy(&self, paths: &Paths, name: &str) {
        let new = resolve_policy(&self.policy, paths, name);
        if let Some(slot) = self.inner.lock().unwrap().get(name) {
            slot.policy.store(new);
        }
    }
}
```

### Daemon RPC

```rust
// daemon/proto.rs
DaemonRequest::ReloadPolicy { name: String }   // -> DaemonResponse::Ok | Error
```

Daemon handler: verify the sandbox dir exists, call
`egress_manager.reload_policy(paths, name)`, reply `Ok`. Idempotent and safe whether
the sandbox is running or stopped.

### Enforce-toggle is hot-safe

The izba CA is baked into **every** guest at boot via the read-only `izba-trust`
virtiofs share (`sandbox.rs:481`, unconditional — not gated on policy). So:

- **bare → enforcing** (turn firewall on): new connections to 80/443 reroute through
  the MITM; the guest already trusts the izba leaf certs → works with no reboot.
  (If the MITM runtime failed to init at daemon start, enforcing fails **closed** per
  existing `router.rs` behavior — correct.)
- **enforcing → bare** (turn firewall off): swap to `AllowAll`; new connections take
  the direct-dial path.

### Tests (TDD)

- `reload_policy` swaps the cell: a slot armed with `AllowAll` then reloaded against a
  written `policy.yaml` denies a now-unlisted host on a fresh `load()` (pure, via the
  `PairListener`/socketpair patterns — no real listener bind).
- `reload_policy` on an unknown/stopped sandbox is a silent `Ok`.
- In-flight snapshot isolation: a connection that already cloned the old policy is
  unaffected by a concurrent `store()` (covered at the `PolicyCell` unit level).
- Proto round-trips `ReloadPolicy`.
- Daemon e2e (KVM-gated): write a deny-all policy, `ReloadPolicy`, assert a previously
  allowed egress is now refused on the next connection without a restart.

## Step 2 — Port-aware policy grammar + Rego

### YAML grammar

`EgressPolicyConfig.allow` becomes `Vec<AllowEntry>`, where an entry is either a bare
host string (back-compat) or a host with explicit ports. `#[serde(untagged)]` keeps
every existing `allow: [<string>...]` file parsing unchanged:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Deserialize)]
#[serde(untagged)]
pub enum AllowEntry {
    /// Bare host → implicit web ports [80, 443].
    Host(String),
    /// Host scoped to an explicit port set (REPLACES the default).
    Scoped { host: String, ports: Vec<u16> },
}
```

```yaml
allow:
  - api.anthropic.com          # implicit [80, 443]
  - host: db.internal
    ports: [5432]              # only 5432
  - host: registry.internal
    ports: [443, 5000]
```

Default-port rule: `ports` absent ⇒ `[80, 443]`; present ⇒ exactly that set.

### Rego data document

`EgressPolicyConfig::to_rego_data_json` emits a host→ports map per sandbox instead of
a host list:

```json
{
  "global_domains": {},
  "sandbox_ports": {
    "<sandbox>": { "api.anthropic.com": [80, 443], "db.internal": [5432] }
  }
}
```

(`global_domains` stays effectively empty for a declared `--policy`, as today; it is
restructured to a host→ports map too for uniformity, with the embedded default list
mapping to `[80, 443]`.)

### Rego

`egress.rego` gains the port predicate:

```rego
default allow := false
dest_name := input.host
dest_name := input.dest if not input.host

# Per-sandbox: host must be listed AND the dialed port allowed for that host.
allow if {
    ports := data.sandbox_ports[input.sandbox][dest_name]
    input.port in ports
}

# Global allow-list (embedded default): web ports only.
allow if {
    ports := data.global_domains[dest_name]
    input.port in ports
}
```

`input.port` is already supplied by `policy::input_json` (`policy.rs:131`) and carried
in `FlowDesc.port` from both the tier-1 `PolicyAdapter` (`mitm_runtime.rs:109`, uses
the `OrigDst` port) and the tier-2 `decide_tier2` flow (`router.rs:234`). So **no
datapath plumbing changes** — only the Rego starts reading the port, and `config.rs`
emits the new data shape.

### How this closes the loophole

- **Tier-1 (80/443, MITM):** a bare entry already authorizes exactly the web ports,
  so normal HTTPS is unaffected. A non-web port never reaches tier-1.
- **Tier-2 (non-web TCP, snooped FQDN):** `decide_tier2` calls `policy.check` with the
  real dialed port. With the port predicate, `github.com:22` is denied unless the
  policy explicitly lists `host: github.com / ports: [..., 22]`. The exfil channel is
  closed by default.

### Migration / compatibility

- Existing string-list `policy.yaml` files parse unchanged and now mean "80/443 only"
  — a **tightening**. This is the intended, documented behavior change. Call it out in
  the M2.1 changelog and the netlog/policy docs.
- `EgressPolicyConfig::default()` (empty allow-list) still compiles to deny-all
  (the fail-closed path in `resolve_policy`).

### Tests (TDD)

- `from_yaml` parses bare strings, `{host, ports}` entries, and a mixed list;
  round-trips via serialize.
- `to_rego_data_json` emits the host→ports map; bare host ⇒ `[80,443]`; scoped host ⇒
  exact set.
- Rego eval: `api.anthropic.com:443` allow; `api.anthropic.com:22` **deny** (the
  loophole, now closed); `db.internal:5432` allow; `db.internal:443` deny.
- `decide_tier2` end-to-end: a snooped allow-listed FQDN on a non-listed port is now
  denied; on a listed port, allowed.
- Back-compat: a legacy string-only file still denies a now-unlisted host and allows a
  listed host on 443.

## Step 3 — Shared helpers + CLI + app P4

### 3a. Core: aggregation helper

`daemon/egress/audit.rs` gains a pure aggregator over parsed records:

```rust
pub struct EndpointSummary {
    pub host: Option<String>,   // resolved name, else None (raw IP)
    pub dest_ip: IpAddr,
    pub port: u16,
    pub tier: Tier,
    pub verdict: Verdict,       // latest (current effective) verdict
    pub allow_count: u64,
    pub deny_count: u64,
    pub first_seen_ms: u64,
    pub last_seen_ms: u64,
    pub last_method: Option<String>,
    pub last_path: Option<String>,
}

/// Group records by (host-or-ip, port), newest first. Pure; testable.
pub fn aggregate(records: impl IntoIterator<Item = AuditRecord>) -> Vec<EndpointSummary>;
```

Key = `(host.clone().unwrap_or(ip.to_string()), port)`. Verdict/method/path come from
the latest record in each group; counts tally allow vs deny; sort by `last_seen_ms`
desc.

### 3b. Core: policy-edit helper

A single mutation point both CLI and app use, keeping grammar concerns in one place:

```rust
// daemon/egress/config.rs (or a new policy_edit.rs)
impl EgressPolicyConfig {
    /// Ensure `host` allows `port` (add host and/or port). Returns whether it changed.
    pub fn allow(&mut self, host: &str, port: u16) -> bool;
    /// Remove `port` from `host` (and the host entirely if it has no ports left).
    pub fn block(&mut self, host: &str, port: u16) -> bool;
    /// Serialize back to canonical policy.yaml text.
    pub fn to_yaml(&self) -> String;
}

/// Load (or default-empty), apply an edit, persist to the sandbox's policy.yaml.
/// Returns the new config so the caller can decide whether to reload.
pub fn edit_policy_file(sandbox_dir: &Path, f: impl FnOnce(&mut EgressPolicyConfig)) -> Result<EgressPolicyConfig>;
```

Edit flow (CLI verb or app button): `edit_policy_file(..)` → if the sandbox is running,
send `ReloadPolicy { name }`. One user action, instant effect.

### 3c. CLI surface

- `izba policy show <sandbox>` — print the effective allow-list (host + ports).
- `izba policy allow <sandbox> <host[:port]>` — add; default port 443 if omitted;
  auto-reload.
- `izba policy block <sandbox> <host[:port]>` — remove; auto-reload.
- `izba policy reload <sandbox>` — manual reload (escape hatch after hand-editing).
- `izba netlog <sandbox> --summary` — the aggregated `EndpointSummary` view (the
  existing line-tail stays the default).

`run --policy <file>` already persists + reloads-on-restart; with Step 1 it can also
live-reload when the sandbox is running. The "Enable firewall (seed from traffic)"
operation is also available headless: `izba policy enable <sandbox>` (build an
allow-list from the sandbox's audit log's currently-allowed endpoints, write it, reload).

### 3d. App P4 (Tauri)

Backend Tauri commands call the core helpers on a fresh `RealDaemon`/`Paths` (same
`spawn_blocking` discipline as P2/P3 — never hold the polling `Mutex<DaemonClient>`
across a blocking call):

- `read_netlog(name) -> Vec<EndpointSummary>` — read `logs/egress-audit.jsonl`, parse,
  `audit::aggregate`. Frontend refreshes on the P3 `LogsView` cadence.
- `policy_show(name) -> EgressPolicyConfig`.
- `policy_allow(name, host, port)` / `policy_block(name, host, port)` —
  `edit_policy_file` + `ReloadPolicy`.
- `policy_enable_from_traffic(name)` — seed the allow-list from the current
  allowed-endpoint summaries, write, reload.

Frontend (new Detail tabs):

- **Netlog tab** — a table of `EndpointSummary` rows (endpoint, port, verdict pill,
  allow/deny counts, last-seen, tier), newest on top, auto-refreshing. Per row:
  **Allow** (shown when currently denied) / **Block** (shown when allowed) → the
  matching command → the row flips on the next refresh.
  - **Raw-IP denied rows**: Allow is **disabled** with a tooltip — "no resolved name;
    allowing a bare IP would defeat the SSRF / DNS-rebind guard." Block is still
    available (it just won't have been allowed under an enforcing policy anyway).
- **Bare sandbox** (non-enforcing): instead of per-row actions, a banner —
  "This sandbox has no firewall · N endpoints observed (all allowed) ·
  **[Enable firewall — allow these N]**". The button calls
  `policy_enable_from_traffic`, which makes the sandbox enforcing without breaking the
  agent; per-row actions then become live.
- **Policy tab** — a structured editor: rows of `host` + `ports` over
  `policy_show`/`edit_policy_file`; Save → reload. (A raw-YAML escape hatch is
  optional; the structured editor is the primary surface.)

### Tests (TDD)

- Core: `aggregate` groups, counts, picks latest verdict, sorts by last-seen; handles
  raw-IP (None host) rows; empty input → empty.
- Core: `allow`/`block`/`to_yaml` round-trips; `block` removes the host when its last
  port goes; `edit_policy_file` creates the file when absent and re-reads correctly.
- Core: "seed from traffic" builds an allow-list containing exactly the
  currently-allowed `(host, port)` endpoints from a fixture audit log.
- App backend: each Tauri command against `FakeDaemon` + a temp `Paths` (no socket
  bind). Frontend: vitest for the netlog table (row actions emit the right command,
  raw-IP Allow disabled, bare-sandbox CTA renders) and the policy editor.

## Risks & edges

- **Mutex vs arc-swap for `PolicyCell`.** Start with `Arc<Mutex<Arc<dyn Policy>>>` (no
  new dep; lock held only for an `Arc` clone). Revisit arc-swap only if contention
  ever shows up — the accept loop is one thread per sandbox, so it won't.
- **Reload race with a connection mid-register.** The MITM `register`-before-connect
  rendezvous is unchanged; a flow that already captured its policy snapshot completes
  under it. New flows get the new one. No new race.
- **"Seed from traffic" staleness.** It snapshots whatever the audit log currently
  shows as allowed — if the agent hasn't exercised an endpoint yet, it won't be
  seeded, and the first post-enable hit will be denied (then one click allows it).
  This is acceptable and matches the interactive model; documented in the CTA copy.
- **Behavior change for existing policies** (string list now means 80/443 only). This
  is the loophole fix; it ships with a clear changelog note and is covered by the
  back-compat test.
- **Windows/WHP parity.** Nothing here touches the VMM datapath or the OpenVMM churn
  invariant — hot-reload swaps an in-process `Arc`, the grammar/rego/aggregation are
  host-side pure logic. Existing KVM + WHP e2e gates carry over; the daemon e2e gains
  the reload assertion.
